**Epic:** EPIC-TD (Resolucao de Debitos Tecnicos)

**Prioridade:** P0/P1
**Agente:** @dev (Dex)

---

## Descricao

O ZmobCRM possui tres monolitos frontend que causam degradacao de performance, impossibilidade de teste e lentidao no desenvolvimento: (1) CRMContext com 922 linhas, ~180 props e 98 componentes dependentes que re-renderizam a cada mudanca de estado, (2) DealCockpitClient com 2.525 linhas intestavel, e (3) tools.ts com 1.648 linhas de SQL inline nas AI tools. Adicionalmente, duplicacao de estado Context API + Zustand, client components no layout protegido impedindo SSR, e ausencia de Server Actions.

Esta story decompooe esses monolitos em modulos menores, testáveis e performáticos.

## Criterios de Aceitacao

### AC-01: DealCockpitClient Decomposto
- **Dado** DealCockpitClient com 2.525 linhas em um unico arquivo
- **Quando** decomposto
- **Entao** existem 6-8 sub-componentes independentes E cada um tem <400 linhas E FocusClient (V1) deletado

### AC-02: CRMContext Eliminado
- **Dado** CRMContext com 922 linhas, ~180 props, 98 importacoes
- **Quando** migracao completa
- **Entao** CRMContext nao existe mais E estado gerenciado por hooks Zustand por dominio E mudanca em hook X nao causa re-render em componentes de hook Y (T-22)

### AC-03: tools.ts Modularizado
- **Dado** tools.ts com 1.648 linhas de SQL inline
- **Quando** refatorado
- **Entao** modulos isolados por dominio (deals, contacts, leads, boards) E cada modulo testavel independentemente

### AC-04: Estado Consolidado
- **Dado** duplicacao Context API + Zustand
- **Quando** consolidado
- **Entao** apenas Zustand como gerenciamento de estado E nenhum Context API para estado de negocio

### AC-05: SSR/Streaming Habilitado
- **Dado** layout protegido todo client-side (`'use client'`)
- **Quando** migrado
- **Entao** layout permite Server Components E streaming funciona nas telas principais

### AC-06: Server Actions
- **Dado** zero Server Actions (tudo via Route Handlers)
- **Quando** introduzidos
- **Entao** formularios principais (deal, contact, lead) usam Server Actions

### AC-07: Skeleton Loading
- **Dado** apenas spinner generico no codebase
- **Quando** skeletons implementados
- **Entao** 4 telas principais mostram skeleton loading em vez de spinner (T-15)

|---|---|---|---|
| Regressao ao decompor CRMContext (98 componentes dependentes) | Alta | Alto | Migrar incrementalmente por dominio (decisions, profile primeiro). Manter CRMContext como facade temporaria |
| Breaking changes no SSR ao migrar layout protegido | Media | Alto | Feature flag para rollback. Testar em staging antes de merge |
| DealCockpitClient decomposicao quebra fluxos de deal | Media | Alto | Testes E2E do fluxo completo (T-21) antes e depois |

| tools.ts refatoracao quebra respostas de IA | Media | Critico | Testes de integracao por modulo. Comparar output antes/depois para mesmas queries |
