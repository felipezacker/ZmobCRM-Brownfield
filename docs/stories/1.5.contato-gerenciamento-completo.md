## User Story

**Como** corretor ou diretor,
**Quero** visualizar e editar todas as informações de um contato (data de nascimento, origem, observações) e acessar um cockpit dedicado do contato,
**Para que** eu tenha uma gestão completa dos meus contatos sem precisar recorrer a ferramentas externas.

---

## Acceptance Criteria

- [ ] AC1: O formulário de criação de contato deve incluir os campos: `name` (obrigatório), `email`, `phone`, `ownerId` (CorretorSelect), `birthDate` (date picker), `source` (select: Website, LinkedIn, Indicação, Manual), `notes` (textarea)
- [ ] AC2: O formulário de edição de contato deve exibir todos os campos do AC1 preenchidos com os valores atuais do contato
- [ ] AC3: O formulário de edição deve exibir como **somente leitura** os campos calculados: `lastInteraction`, `lastPurchaseDate`, `totalValue` (formatado como moeda BRL)
- [ ] AC4: Os campos do formulário devem ser organizados em seções visuais: "Dados Básicos" (name, email, phone, ownerId), "Informações Pessoais" (birthDate, source), "Observações" (notes), e "Histórico" (campos read-only, apenas na edição)
- [ ] AC5: O modal de edição de contato deve conter um botão de cockpit (ícone `Maximize2`) no header, seguindo o mesmo padrão visual do `DealDetailModal`, que navega para `/contacts/{contactId}/cockpit`
- [ ] AC6: A rota `/contacts/[contactId]/cockpit` deve existir e renderizar uma página de cockpit do contato (scaffold inicial com dados básicos do contato)
- [ ] AC7: Os campos `stage` e `status` **não** devem aparecer no formulário (mantêm gestão via tabs e inline na lista)
- [ ] AC8: O campo `birthDate` deve aceitar datas válidas e salvar no formato ISO (YYYY-MM-DD)
- [ ] AC9: O campo `source` deve usar um select com as opções: `WEBSITE`, `LINKEDIN`, `REFERRAL`, `MANUAL`
- [ ] AC10: Todos os novos campos devem ser persistidos corretamente no banco via `contactsService.create()` e `contactsService.update()`

## Scope

### IN Scope
- Expansão do `ContactFormModal.tsx` com novos campos (birthDate, source, notes)

- Stage/status no formulário (mantém tabs/inline)
- Sistema de tags
- Editor rico de notes (textarea simples)
- Conteúdo completo do cockpit (será story futura)

### Campos já suportados no DB e types
A interface `Contact` em `types/types.ts` já possui `birthDate`, `source`, `notes`, `lastInteraction`, `lastPurchaseDate`, `totalValue`. O `contactsService` em `lib/supabase/contacts.ts` já faz o mapeamento snake_case → camelCase via `transformContact`. Não há necessidade de alterações no service layer ou types — o gap é **puramente na UI e controller**.

### formData atual no controller
O `useContactsController.ts` inicializa formData com: `name`, `email`, `phone`. No submit, envia apenas esses campos + `ownerId`. Precisa expandir para incluir `birthDate`, `source`, `notes`.
