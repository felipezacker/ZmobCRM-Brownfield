### PadrГЈo de autorizaГ§ГЈo atual

O sistema usa `profile?.role === 'admin'` / `profile.role !== 'admin'` como guard binГЎrio. Com 3 nГ­veis, recomenda-se criar um helper:

```typescript
// lib/auth/roles.ts
const ROLE_HIERARCHY = { admin: 3, diretor: 2, corretor: 1 } as const;
type Role = keyof typeof ROLE_HIERARCHY;

export function hasMinRole(userRole: Role, requiredRole: Role): boolean {
  return ROLE_HIERARCHY[userRole] >= ROLE_HIERARCHY[requiredRole];
}

// Uso: hasMinRole(profile.role, 'diretor') в†’ true para admin e diretor
```

### RLS вЂ” LГіgica para `diretor`

Nas policies de deals/contatos, o `diretor` deve ter visibilidade de toda a organizaГ§ГЈo (similar ao admin), mas NГѓO ter permissГЈo de DELETE em dados sensГ­veis nem acesso a tabelas de configuraГ§ГЈo.

**Estado atual das policies de deals/contacts:** `USING(true)` вЂ” qualquer autenticado vГЄ tudo. Para implementar AC4 (diretor vГЄ tudo, corretor vГЄ sГі os seus), Г© necessГЎrio **substituir** essas policies por:
```sql
-- Corretor: apenas seus deals/contacts
CREATE POLICY "Users can view own records" ON public.deals
  FOR SELECT TO authenticated
  USING (
    owner_id = auth.uid()
    OR auth.uid() IN (
      SELECT id FROM public.profiles
      WHERE organization_id = deals.organization_id
        AND role IN ('admin', 'diretor')
    )
  );
```

### 5. Gate Decision

**Verdict: CONCERNS**

**Rationale:** Implementation is functionally correct and secure at the API/DB layer.
