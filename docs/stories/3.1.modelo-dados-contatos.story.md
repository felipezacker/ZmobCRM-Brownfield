## Story

**As a** corretor imobiliario,
**I want** o modelo de contatos expandido com campos especificos do mercado imobiliario (CPF, classificacao, temperatura, endereco, multiplos telefones),
**so that** eu possa registrar informacoes essenciais dos meus clientes sem precisar de campos customizados ou anotacoes manuais.

## Acceptance Criteria

1. `contacts` possui novos campos: `cpf` (TEXT, unique parcial por org), `contact_type` (PF/PJ, default PF), `classification` (COMPRADOR/VENDEDOR/LOCATARIO/LOCADOR/INVESTIDOR/PERMUTANTE), `temperature` (HOT/WARM/COLD, default WARM), `address_cep`, `address_city`, `address_state` (UF 2 chars), `profile_data` (JSONB)
2. Tabela `contact_phones` existe com: `id`, `contact_id` FK, `phone_number`, `phone_type` (CELULAR/COMERCIAL/RESIDENCIAL), `is_whatsapp`, `is_primary`, `organization_id` FK, `created_at`, `updated_at`
3. Unique index parcial impede CPF duplicado dentro da mesma org (permite NULL)
4. Unique index parcial garante apenas 1 telefone primario por contato
5. Index de busca por telefone existe para dedup: `idx_contact_phones_number_org(organization_id, phone_number)`
6. RLS de `contact_phones` segue padrao org-level (todos da org veem/editam)
7. Indexes de filtro existem: classification, temperature, contact_type (todos compostos com org_id)
8. Index trigram (`pg_trgm`) existe para busca server-side por nome
9. Index por email+org existe para dedup por email
10. Triggers `updated_at` existem em `contacts` e `deals` (ambas faltavam)
11. Formulario de contato exibe e permite editar todos os novos campos
12. Lista de contatos exibe as novas colunas (classificacao, temperatura)
13. Validacao de CPF no frontend (mascara + digitos verificadores)
14. Validacao de CEP no frontend (mascara 00000-000)

### Risk Assessment

- **Risco funcional (Issue #1)**: Usuarios vao adicionar telefones no formulario achando que estao salvando, mas os dados sao perdidos no submit. Impacto medio — campo `phone` legacy continua funcionando para 1 telefone.
- **Seguranca**: RLS org-level correta em contact_phones. CHECK constraints previnem dados invalidos.
- **Performance**: 17 indexes criados, pg_trgm para busca fuzzy. Adequado para o volume atual.

### Verdict Rationale

CONCERNS devido ao Issue #1 (HIGH): telefones adicionados via UI nao persistem na tabela contact_phones. O campo legacy `phone` em contacts funciona (1 telefone), entao nao e blocking para o usuario basico, mas a funcionalidade multi-telefone prometida pelo AC2 nao esta operacional end-to-end. Recomendo corrigir antes do merge ou criar story de follow-up com prioridade alta.
