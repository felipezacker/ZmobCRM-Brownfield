## Story

**As a** corretor imobiliario,
**I want** uma visao 360 de cada contato com timeline de atividades, deals vinculados, chat IA, notas e preferencias de imovel num layout de cockpit,
**so that** eu tenha todas as informacoes do cliente num unico lugar para tomar decisoes rapidas e oferecer atendimento personalizado.

## Acceptance Criteria

1. Cockpit do contato usa layout 3 colunas (mesmo padrao do cockpit de deals: left 360px, center flex, right 420px)
2. **Left rail** exibe: temperatura visual (badge grande), dados completos (CPF, endereco, telefones, classificacao, tipo PF/PJ, source, email), perfil de interesse (contact_preferences resumido)
3. **Center** exibe: timeline unificada de atividades do contato (via deals vinculados) em ordem cronologica
4. **Center** permite: adicionar nota rapida inline (input + botao, salva em contact_notes)
5. **Right rail** tem 3 tabs: "IA" (chat contextualizado), "Deals" (mini-cards dos deals vinculados), "Notas" (CRUD de notas do contato)
6. Tab "Deals" exibe mini-cards com: titulo, valor (R$), stage badge, tipo negocio, probabilidade
7. Tab "IA" recebe snapshot do contato (dados + deals + preferencias) como contexto para o chat
8. Tab "Notas" permite criar, editar e deletar notas do contato (tabela `contact_notes` ou via deal_notes com filtro)
9. Header do cockpit exibe: avatar, nome, classificacao badge, temperatura badge, estagio badge, botao editar (abre ContactFormModal)
10. Pipeline bar no topo: exibe stage do contato (LEAD → MQL → PROSPECT → CUSTOMER) com indicador visual
11. Botao "Voltar para lista" funciona corretamente
12. Cockpit carrega dados do contato, deals, atividades e preferencias em paralelo (sem waterfall)

## Dev Notes

### Source: Codebase existente — Deal Cockpit como referencia

**Pattern a seguir:** `features/deals/cockpit/` com 12 arquivos:
- `DealCockpitClient.tsx` — orquestrador (711 linhas)
- `cockpit-types.ts`, `cockpit-utils.ts`, `cockpit-ui.tsx` — shared
- `CockpitPipelineBar.tsx`, `CockpitHealthPanel.tsx`, `CockpitNextActionPanel.tsx`
- `CockpitDataPanel.tsx`, `CockpitTimeline.tsx`, `CockpitChecklist.tsx`
- `CockpitRightRail.tsx`, `TemplatePickerModal.tsx`

**Scaffold existente:**
- Rota: `app/(protected)/contacts/[contactId]/cockpit/page.tsx` — renderiza ContactCockpitClient
- Componente: `features/contacts/cockpit/ContactCockpitClient.tsx` — stub com 5 campos basicos + placeholder

**Componentes reutilizaveis do deal cockpit:**
- `Panel`, `TabButton`, `Chip` de `cockpit-ui.tsx`
- `formatCurrencyBRL`, `formatAtISO` de `cockpit-utils.ts`
- `UIChat` para tab IA

**Dados necessarios (queries paralelas):**
1. Contato: `contacts WHERE id = contactId` (com novos campos)
2. Telefones: `contact_phones WHERE contact_id = contactId`
3. Preferencias: `contact_preferences WHERE contact_id = contactId`
4. Deals: `deals WHERE contact_id = contactId AND deleted_at IS NULL`
5. Atividades: `activities WHERE deal_id IN (deals do contato)` — para timeline

**Stages do contato (existentes):**
- LEAD, MQL, PROSPECT, CUSTOMER (definidos no enum do sistema)

### Decisao arquitetural: Notas do contato

**Opcao A (recomendada):** Reutilizar tabela `activities` com `type = 'NOTE'`. Activities ja tem `deal_id` que referencia deals vinculados ao contato. Para notas "avulsas" do contato (sem deal), pode-se usar o primeiro deal ativo ou criar uma referencia direta.

**Opcao B:** Criar tabela `contact_notes` (similar a deal_notes). Mais limpo mas requer migration.

**Decisao para o @dev:** Escolher a opcao mais pragmatica. Se Opcao A nao cobrir o caso de contato sem deals, usar Opcao B.

### Riscos
- Contato sem deals: timeline vazia, tab Deals vazia — exibir empty states adequados
- Performance: muitos deals com muitas atividades pode gerar query pesada — limitar a ultimas 50 atividades
- AI snapshot pode ser grande — limitar a ultimos 5 deals e 10 atividades

### Testing
- **Foco:** Layout responsivo, timeline carrega corretamente, tabs funcionam, IA recebe contexto
- **Edge cases:** Contato sem deals, contato sem preferencias, contato sem atividades
