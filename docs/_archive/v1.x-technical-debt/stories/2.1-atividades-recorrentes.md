## Story

**As a** usuario do CRM,
**I want** criar atividades que se repetem (diario, semanal, mensal) e transformar existentes em recorrentes,
**so that** nao precise recriar manualmente atividades rotineiras, ganhando produtividade no acompanhamento de deals.

## Descricao

Hoje cada atividade e um evento unico. O usuario quer criar atividades que se repetem (diario, semanal, mensal) e transformar existentes em recorrentes. Referencia: Google Calendar.

**Comportamento-chave:**
- **Lista:** mostra apenas a proxima ocorrencia pendente (row real no DB)
- **Calendario:** projeta TODAS as ocorrencias futuras da semana visivel (instancias virtuais calculadas client-side, estilo Google Calendar)

## Criterios de Aceitacao

### AC-01: Criar Atividade Recorrente
- **Dado** o formulario de nova atividade aberto
- **Quando** seleciono recorrencia "Semanalmente" e data limite "2026-03-31"
- **Entao** a atividade e criada com `recurrence_type = 'weekly'` e `recurrence_end_date = '2026-03-31'`

### AC-02: Badge na Lista
- **Dado** uma atividade recorrente na lista
- **Quando** visualizo a lista de atividades
- **Entao** aparece badge azul com icone Repeat e texto "Diario"/"Semanal"/"Mensal" ao lado do titulo

### AC-03: Projecao no Calendario
- **Dado** uma atividade recorrente semanal criada em 2026-02-25 (terca)
- **Quando** navego para a semana de 2026-03-02 no calendario
- **Entao** vejo a instancia projetada na terca 2026-03-04 com visual diferenciado (opacidade 70%, borda tracejada, icone Repeat)

### AC-04: Completar Gera Proxima
- **Dado** uma atividade recorrente semanal com data 2026-02-25
- **Quando** marco como concluida
- **Entao** a atividade atual fica concluida E uma nova atividade e criada com data 2026-03-04 (mesma hora, mesmo deal, mesma recorrencia)

### AC-05: Respeitar Data Limite
- **Dado** uma atividade recorrente com `recurrence_end_date = '2026-03-01'`
- **Quando** marco como concluida e a proxima data seria 2026-03-08
- **Entao** a atividade e concluida SEM criar proxima E o calendario NAO projeta alem de 2026-03-01

### AC-06: Transformar Existente em Recorrente
- **Dado** uma atividade existente sem recorrencia
- **Quando** edito e seleciono recorrencia "Mensalmente"
- **Entao** a atividade ganha `recurrence_type = 'monthly'` E o calendario passa a projetar ocorrencias futuras

### AC-07: Card Projetado Clicavel
- **Dado** uma instancia projetada no calendario
- **Quando** clico nela
- **Entao** abre o formulario pre-preenchido com dados da atividade-pai para criar a instancia real

### AC-08: Snooze de Recorrente
- **Dado** uma atividade recorrente
- **Quando** uso snooze (adiar)
- **Entao** a data da instancia atual e adiada mas a regra de recorrencia NAO e alterada

### AC-09: Validacao de Datas
- **Dado** o formulario com recorrencia ativa
- **Quando** seleciono uma data limite anterior a data da atividade
- **Entao** o formulario mostra erro de validacao e impede o submit

## Escopo

### IN

- Tipos TypeScript: `RecurrenceType` + campos na interface `Activity`

- Controller: logica de recorrencia no complete, form data, handlers
- Form UI: select de recorrencia + input de data limite
- Badge na lista: icone + texto de recorrencia
- Calendario: projecao client-side de instancias virtuais

### OUT
- Recorrencia customizada (a cada N dias, dias especificos da semana)
- Series linkadas (parentActivityId, editar todas as instancias futuras)
- Edicao em lote de recorrentes ("alterar esta e todas as futuras")
- Notificacoes/lembretes de atividades
- Recorrencia por hora (apenas dia)
- Sync com Google Calendar

## Riscos

| Risco | Probabilidade | Impacto | Mitigacao |
|---|---|---|---|
| Race condition: complete + create sao 2 chamadas separadas. Se create falha, cadeia de recorrencia quebra | Media | Alto | Implementar try/catch: se create falhar, reverter o complete via toggleCompletion. Logar erro. Toast de falha. |
| Performance: muitas atividades recorrentes geram muitas instancias virtuais no calendario | Baixa | Medio | Limitar projecao a 4 semanas no futuro. useMemo com deps corretas. |
| UX confusa: usuario nao entende diferenca entre instancia real e projetada | Media | Medio | Visual claramente diferenciado (opacidade 70%, borda tracejada). Tooltip "Clique para criar esta atividade". |
| ID de instancias projetadas conflita com logica que espera UUID | Baixa | Alto | Usar prefixo `projected-` no id. Guardar em campo `_isProjected: true` no objeto. Filtrar projetadas de qualquer operacao de DB. |

## Valor de Negocio

- Reduz trabalho manual de recriar atividades repetitivas (follow-ups semanais, calls mensais)
- Alinha CRM com expectativa de mercado (Google Calendar, HubSpot, Pipedrive tem recorrencia)
- Melhora disciplina de acompanhamento: proxima ocorrencia criada automaticamente

## Edge Cases

| Cenario | Comportamento Esperado |
|---|---|
| End date atingida | Nao projeta alem, toast ao concluir ultima |
| Editar recorrente | Muda so a instancia atual (sem parentActivityId) |
| Deletar recorrente | Deleta so a instancia (sem serie) |
| Reabrir (uncomplete) | Nao desfaz proxima ja criada |
| Bulk complete | Cada item passa por handleToggleComplete (ativa recorrencia) |
| Card projetado no calendario | Clicavel, abre form pre-preenchido |
| Snooze de recorrente | Adia instancia, recorrencia intacta |
| Mes com menos dias (monthly: 31 jan → 28 fev) | Clamp para ultimo dia do mes |

### Mapeamento Form ↔ DB (CRITICO)

```
Form: recurrenceType = 'none'  →  API/DB: recurrence_type = NULL
Form: recurrenceType = 'daily' →  API/DB: recurrence_type = 'daily'
Form: recurrenceEndDate = ''   →  API/DB: recurrence_end_date = NULL
```

### Review @data-engineer (Dara)

**RLS:** Zero impacto — policies filtram por `organization_id`/`owner_id`, nao tocam recurrence.
**Triggers:** Zero impacto — `cascade_soft_delete_activities_by_contact` nao afetado.
**Realtime:** Zero impacto — colunas novas aparecem automaticamente no payload.
**Rows existentes:** Zero impacto — DEFAULT NULL, nenhum backfill necessario.

**Constraint cruzada adicionada:** `chk_recurrence_end_date_requires_type` impede `recurrence_end_date` preenchida sem `recurrence_type` (defesa em profundidade).

**Index parcial adicionado:** `idx_activities_recurrence` cobre queries de recorrencia sem impactar inserts normais (WHERE recurrence_type IS NOT NULL).

**Tech debt registrado (nao bloqueante):** Index composto `(organization_id, date DESC)` para otimizar `getAll()` + RLS. Criar em story futura.

**Enhancement futuro (nao bloqueante):** RPC `complete_and_spawn_next(activity_id)` para executar complete + create em transacao server-side. Hoje mitigado via try/catch no controller.

**Timestamp da migration:** Nome `20260226000000` esta sequencial apos ultima migration `20260225000000`. Se implementar em data diferente, ajustar timestamp.

### Retry Fallback Pattern (existente)

O `activitiesService.create` ja tem retry fallback para `participant_contact_ids`. Os novos campos devem seguir o mesmo pattern: tentar com os campos, se falhar (coluna inexistente), retry sem eles.
