## User Story

**Como** administrador de uma imobiliária,
**Quero** que o sistema tenha os níveis "corretor" (no lugar de "vendedor") e um novo nível "diretor" com visibilidade gerencial,
**Para que** a nomenclatura reflita o mercado imobiliário e eu possa delegar gestão de equipe sem dar acesso total de admin.

### Padrão de autorização atual

O sistema usa `profile?.role === 'admin'` / `profile.role !== 'admin'` como guard binário. Com 3 níveis, recomenda-se criar um helper:

```typescript
// lib/auth/roles.ts
const ROLE_HIERARCHY = { admin: 3, diretor: 2, corretor: 1 } as const;
type Role = keyof typeof ROLE_HIERARCHY;

export function hasMinRole(userRole: Role, requiredRole: Role): boolean {
  return ROLE_HIERARCHY[userRole] >= ROLE_HIERARCHY[requiredRole];
}

// Uso: hasMinRole(profile.role, 'diretor') → true para admin e diretor
```

### RLS — Lógica para `diretor`

Nas policies de deals/contatos, o `diretor` deve ter visibilidade de toda a organização (similar ao admin), mas NÃO ter permissão de DELETE em dados sensíveis nem acesso a tabelas de configuração.

**Estado atual das policies de deals/contacts:** `USING(true)` — qualquer autenticado vê tudo. Para implementar AC4 (diretor vê tudo, corretor vê só os seus), é necessário **substituir** essas policies por:
```sql
-- Corretor: apenas seus deals/contacts
CREATE POLICY "Users can view own records" ON public.deals
  FOR SELECT TO authenticated
  USING (
    owner_id = auth.uid()
    OR auth.uid() IN (
      SELECT id FROM public.profiles
      WHERE organization_id = deals.organization_id
        AND role IN ('admin', 'diretor')
    )
  );
```
