## Story

**As a** corretor imobiliario,
**I want** registrar o perfil de interesse de cada contato (tipo de imovel, faixa de preco, regioes, quartos, urgencia),
**so that** eu possa filtrar e oferecer imoveis adequados ao perfil do cliente no futuro (matching).

## Acceptance Criteria

1. Tabela `contact_preferences` existe com: `id` UUID PK, `contact_id` FK, `property_types` TEXT[], `purpose` (MORADIA/INVESTIMENTO/VERANEIO), `price_min` NUMERIC, `price_max` NUMERIC, `regions` TEXT[], `bedrooms_min` INTEGER, `parking_min` INTEGER, `area_min` NUMERIC, `accepts_financing` BOOLEAN, `accepts_fgts` BOOLEAN, `urgency` (IMMEDIATE/3_MONTHS/6_MONTHS/1_YEAR), `notes` TEXT, `organization_id` FK, `created_at`, `updated_at`
2. CHECK constraint garante `price_min <= price_max` (quando ambos preenchidos)
3. Index parcial existe para matching por faixa de preco: `idx_contact_prefs_price_range`
4. RLS de `contact_preferences` segue padrao org-level (mesma politica de contact_phones)
5. Trigger `updated_at` existe na tabela
6. Um contato pode ter multiplos perfis de interesse (1:N) — ex: "apto para morar" + "terreno para investir"
7. UI permite criar, editar e deletar perfis de interesse na tela do contato
8. Formulario de preferencias tem campos com tipos adequados (multiselect para property_types, range para preco, multiselect para regioes)
9. COMMENT ON documenta valores esperados de property_types

### Issues

| # | Severity | Category | Description | Recommendation |
|---|----------|----------|-------------|----------------|
| 1 | **MEDIUM** | code | **`false \|\| null` bug em formToPayload**: Linhas 99-100 de ContactPreferencesSection.tsx usam `form.acceptsFinancing \|\| null` e `form.acceptsFgts \|\| null`. Quando o usuario DESMARCA o checkbox (valor `false`), `false \|\| null` avalia para `null`. O DB recebe NULL ao inves de FALSE, perdendo a escolha explicita "nao aceita financiamento/FGTS". | Trocar para `form.acceptsFinancing ?? null` (nullish coalescing) ou simplesmente `form.acceptsFinancing` em ambas as linhas. |
| 2 | LOW | ux | **Delete sem confirmacao**: handleDelete (linha 516) exclui perfil imediatamente sem dialogo de confirmacao. | Adicionar confirm() ou modal de confirmacao para evitar exclusao acidental. Tech debt UX. |

### Risk Assessment

- **Issue #1 e um bug logico real**: Impacta a funcionalidade de matching futuro (Story 3.5+). Se o corretor registra que o cliente NAO aceita financiamento, essa informacao e perdida como NULL (indeterminado). Correcao e trivial (1 caractere: `||` → `??`).
- **Seguranca**: RLS corretas. Nao ha SQL injection risk (Supabase client SDK parametriza queries).
- **Padrao de save imediato**: Preferencias operam com save direto (nao via form state do modal pai) — padrao limpo para entidade 1:N complexa.

### Verdict Rationale

CONCERNS: Issue #1 (MEDIUM) e um bug logico onde `false || null` perde a intencao explicita do usuario em checkboxes de financiamento/FGTS. Correcao e de 1 linha. A funcionalidade principal de CRUD opera corretamente. Recomendo fix rapido antes do merge.
